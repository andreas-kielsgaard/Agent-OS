# Core Extraction Inventory

## Purpose

This inventory records the first opinionated extraction plan for reusable Agent OS core files before any Field Platform core files are copied into this repository.

Field Platform is the reference installation. It demonstrates a working Agent OS installation, but it is not the universal template. The classifications below start from the current Field Platform files, then separate reusable core guidance from adapter-owned project binding and plugin-owned evidence production.

## Required Neutralization

Before copying adapted files into core, remove or generalize these Field Platform-specific assumptions:

- Field Platform-specific project-decision routing: core may say that human-owned project decisions can exist, but the adapter must declare where local project-control maps and decision surfaces live.
- Local command names and active replacement tools: replace `change-surface`, `test-selection`, `change-verification`, `repo-health`, `depcruise:active-source`, `corepack pnpm ...`, and `agent-os context ...` with adapter-declared active tools or plugin/evidence-producer references.
- Generated/archive policy specifics: core should define that generated and archive surfaces are evidence or historical context, while adapters own the concrete paths, exclusions, and protected surfaces.
- Project-control file assumptions: core must not require `Agent OS/project-control-files/project-setup-map.md` or any Field Platform folder layout. Install/adapters own those paths.
- Stack and layout references: neutralize references that imply TypeScript, pnpm, React, Drizzle, dependency-cruiser, monorepo layout, shared UI, web routes, Storybook, Vitest, Playwright, Biome, Zod, or Field Platform domain concepts are universal.
- Tool authority language: preserve the rule that generated output and tool output are evidence only, but route specific evidence sources through adapters and plugin availability.

## Core Boundary Rules

- Core owns attention/routing/execution discipline and evidence-vs-authority rules.
- Adapters own project-specific roots, source policy, command profile, validation profile, project-control routing, and enabled evidence producers.
- Plugins and evidence producers own deterministic evidence integrations and their disabled, unavailable, failed, and successful behavior.
- Install instructions own the agent-executable process for applying Agent OS to a repository.

## Proposed First Copy Batch

Recommended first copy batch:

1. Adapted `Agent OS/agent-os-bootloader.md` to `core/agent-os-bootloader.md`.
2. Adapted `Agent OS/agent-os-execution-instructions.md` to `core/agent-os-execution-instructions.md`.
3. `Agent OS/prompt-files/agent-attention-system/agent-attention-system-usage.md` to `core/agent-attention-system/agent-attention-system-usage.md`.
4. `Agent OS/prompt-files/review-checklist.md` to `core/review-checklist.md`.
5. `Agent OS/prompt-files/agent-attention-system/maps/lens-map.md` to `core/agent-attention-system/maps/lens-map.md`.

This batch establishes the reusable boot sequence, execution posture, context-loading discipline, review cues, and lens inventory without copying the larger mode, behavior, skill, or tool surface too early.

Files intentionally left out of the first copy batch:

- `task-mode-map.md`: useful routing structure, but it currently mixes generic modes with UI-facing, Drizzle/Zod, dependency-boundary, route/page, and project-decision assumptions. Extract after deciding whether UI-facing modes are core or a mode pack.
- `behavior-map.md`: reusable taxonomy, but it names dependency-cruiser, active replacement tools, package scripts, and project decisions. Extract after adapter/plugin evidence wording is settled.
- `skill-map.md`: valuable skill routing, but it binds skills to current command names, Drizzle, shared UI, and dependency-cruiser. Extract after skill destination and active-tool surfacing are decided.
- `tool-map.md`: not core as-is. It is an installed-project tool map and should become an adapter-owned file or install template only after adapter schema and plugin contracts exist.
- `change-impact-checklists.md`: useful checklist shape, but it currently embeds shared UI, dependency-cruiser, active replacement tools, route/page assumptions, and project-decision references.

- How are adapter-declared active tools surfaced to core prompts without hardcoding command names?
- Where do root `AGENTS.md` templates belong: install templates, adapter templates, or a separate root-entry contract package?
- How much context tooling should become a reusable package, and how much should remain project-local tooling selected by adapters?
